      32             : /*!
      33             :  * \brief A V-cycle geometric multgrid solver for linear equations \f$Ax = b\f$
      34             :  *
      35             :  * This linear solver iteratively corrects an initial guess \f$x_0\f$ by
      36             :  * restricting the residual \f$b - Ax\f$ to a series of coarser grids, solving
      37             :  * for a correction on the coarser grids, and then prolongating (interpolating)
      38             :  * the correction back to the finer grids. The solves on grids with different
      39             :  * scales can very effectively solve large-scale modes in the solution, which
      40             :  * Krylov-type linear solvers such as GMRES or Conjugate Gradients typically
      41             :  * struggle with. Therefore, a multigrid solver can be an effective
      42             :  * preconditioner for Krylov-type linear solvers (see
      43             :  * `LinearSolver::gmres::Gmres` and `LinearSolver::cg::ConjugateGradient`). See
      44             :  * \cite Briggs2000jp for an introduction to multigrid methods.
      45             :  *
      46             :  * \par Grid hierarchy
      47             :  * This geometric multigrid solver relies on a strategy to coarsen the
      48             :  * computational grid in a way that removes small-scale modes. We currently
      49             :  * h-coarsen the domain, meaning that we create multigrid levels by successively
      50             :  * combining two elements into one along every dimension of the grid. We only
      51             :  * p-coarsen the grid in the sense that we choose the smaller of the two
      52             :  * polynomial degrees when combining elements. This strategy follows
      53             :  * \cite Vincent2019qpd. See `LinearSolver::multigrid::ElementsAllocator` and
      54             :  * `LinearSolver::multigrid::coarsen` for the code that creates the multigrid
      55             :  * hierarchy.
      56             :  *
      57             :  * \par Inter-mesh operators
      58             :  * The algorithm relies on operations that project data between grids. Residuals
      59             :  * are projected from finer to coarser grids ("restriction") and solutions are
      60             :  * projected from coarser to finer grids ("prolongation"). We use the standard
      61             :  * \f$L_2\f$-projections implemented in
      62             :  * `Spectral::projection_matrix_child_to_parent` and
      63             :  * `Spectral::projection_matrix_parent_to_child`, where "child" means the finer
      64             :  * grid and "parent" means the coarser grid. Note that the residuals \f$b -
      65             :  * Ax\f$ may or may not already include a mass matrix and Jacobian factors,
      66             :  * depending on the implementation of the linear operator \f$A\f$. To account
      67             :  * for this, provide a tag as the `ResidualIsMassiveTag` that holds a `bool`.
      68             :  * See section 3 in \cite Fortunato2019jl for details on the inter-mesh
      69             :  * operators.
      70             :  *
      71             :  * \par Smoother
      72             :  * On every level of the grid hierarchy the multigrid solver relies on a
      73             :  * "smoother" that performs an approximate linear solve on that level. The
      74             :  * smoother can be any linear solver that solves the `smooth_fields_tag` for the
      75             :  * source in the `smooth_source_tag`. Useful smoothers are asynchronous linear
      76             :  * solvers that parallelize well, such as `LinearSolver::Schwarz::Schwarz`. The
      77             :  * multigrid algorithm doesn't assume anything about the smoother, but requires
      78             :  * only that the `PreSmootherActions` and the `PostSmootherActions` passed to
      79             :  * `solve` leave the `smooth_fields_tag` in a state that represents an
      80             :  * approximate solution to the `smooth_source_tag`, and that
      81             :  * `db::add_tag_prefix<LinearSolver::Tags::OperatorAppliedTo,
      82             :  * smooth_fields_tag>` is left up-to-date as well.
      83             :  * The smoother can assume that, on entry, these two tags represent the initial
      84             :  * fields and the linear operator applied to the initial fields, respectively.
      85             :  * Here's an example of setting up a smoother for the multigrid solver:
      86             :  *
      87             :  * \snippet Test_MultigridAlgorithm.cpp setup_smoother
      88             :  *
      89             :  * The smoother can be used to construct an action list like this:
      90             :  *
      91             :  * \snippet Test_MultigridAlgorithm.cpp action_list
      92             :  *
      93             :  * \par Algorithm overview
      94             :  * Every iteration of the multigrid algorithm performs a V-cycle over the grid
      95             :  * hierarchy. One V-cycle consists of first "going down" the grid hierarchy,
      96             :  * from the finest to successively coarser grids, smoothing on every level
      97             :  * ("pre-smoothing", can be controlled by the option
      98             :  * `LinearSolver::multigrid::Tags::EnablePreSmoothing`),
      99             :  * and then "going up" the grid hierarchy again, smoothing on
     100             :  * every level again ("post-smoothing"). When going down, the algorithm projects
     101             :  * the remaining residual of the smoother to the next-coarser grid, setting it
     102             :  * as the source for the smoother on the coarser grid. When going up again, the
     103             :  * algorithm projects the solution of the smoother to the next-finer grid,
     104             :  * adding it to the solution on the finer grid as a correction. The bottom-most
     105             :  * coarsest grid (the "tip" of the V-cycle) may skip the post-smoothing, so the
     106             :  * result of the pre-smoother is immediately projected up to the finer grid
     107             :  * (controlled by the
     108             :  * `LinearSolver::multigrid::Tags::EnablePostSmoothingAtBottom` option). On the
     109             :  * top-most finest grid (the "original" grid that represents the overall
     110             :  * solution) the algorithm applies the smoothing and the corrections from the
     111             :  * coarser grids directly to the solution fields.
     112             :  */
