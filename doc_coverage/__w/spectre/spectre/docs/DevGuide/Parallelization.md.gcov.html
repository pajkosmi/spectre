       9             : # Introduction {#dev_guide_parallelization_introduction}
      10             : 
      11             : SpECTRE builds a layer on top of Charm++ that performs various safety checks and
      12             : initialization for the user that can otherwise lead to difficult-to-debug
      13             : undefined behavior. The central concept is what is called a %Parallel
      14             : Component. A %Parallel Component is a struct with several type aliases that
      15             : is used by SpECTRE to set up the Charm++ chares and allowed communication
      16             : patterns. %Parallel Components are input arguments to the compiler, which then
      17             : writes the parallelization infrastructure that you requested for the executable.
      18             : There is no restriction on the number of %Parallel Components, though
      19             : practically it is best to have around 10 at most.
      20             : 
      21             : Here is an overview of what is described in detail in the sections below:
      22             : 
      23             : - \ref dev_guide_parallelization_metavariables_class "Metavariables": Provides
      24             :   high-level configuration to the compiler, e.g. the physical system to be
      25             :   simulated.
      26             : - \ref dev_guide_parallelization_phases_of_execution "Phases": Defines distinct
      27             :   simulation phases separated by a global synchronization point,
      28             :   e.g. `Initialization`, `Evolve` and `Exit`.
      29             : - \ref dev_guide_parallelization_core_algorithm "Algorithm": In each phase,
      30             :   repeatedly iterates over a list of actions until the current phase ends.
      31             : - \ref dev_guide_parallelization_parallel_components "Parallel component":
      32             :   Maintains and executes its algorithm.
      33             : - \ref dev_guide_parallelization_actions "Action": Performs a computational
      34             :   task, e.g. evaluating the right hand side of the time evolution equations. May
      35             :   require data to be received from another action potentially being executed on
      36             :   a different core or node.
      37             : 
      38             : # The Metavariables Class {#dev_guide_parallelization_metavariables_class}
      39             : 
      40             : SpECTRE takes a different approach to input options passed to an executable than
      41             : is common. SpECTRE not only reads an input file at runtime but also has many
      42             : choices made at compile time. The compile time options are specified by what is
      43             : referred to as the metavariables. What exactly the metavariables struct
      44             : specifies depends on the executable, but all metavariables structs must
      45             : specify the following:
      46             : 
      47             : - `help`: a `static constexpr Options::String` that will be printed as part of
      48             :   the help message. It should describe the executable and basic usage of it, as
      49             :   well as any non-standard options that must be specified in the metavariables
      50             :   and their current values. An example of a help string for one of the testing
      51             :   executables is:
      52             :   \snippet Test_AlgorithmCore.cpp help_string_example
      53             : - `component_list`: a `tmpl::list` of the parallel components (described below)
      54             :   that are to be created. Most evolution executables will have the
      55             :   `DgElementArray` parallel component listed. An example of a `component_list`
      56             :   for one of the test executables is:
      57             :   \snippet Test_AlgorithmCore.cpp component_list_example
      58             : - `using const_global_cache_tags`: a `tmpl::list` of tags that are
      59             :   used to place const items in the GlobalCache.  The alias may be
      60             :   omitted if the list is empty.
      61             : - `using mutable_global_cache_tags`: a `tmpl::list` of tags that are
      62             :   used to place mutable items in the GlobalCache.  The alias may be
      63             :   omitted if the list is empty.
      64             : - `default_phase_order`: an array of Parallel::Phase that must contain at least
      65             :   `Initialization` as the first element and `Exit`as the last element. Phases
      66             :   are described in the next section.
      67             : 
      68             : There are also several optional members:
      69             : 
      70             : - `input_file`: a `static constexpr Options::String` that is the default name of
      71             :   the input file that is to be read. This can be overridden at runtime by
      72             :   passing the `--input-file` argument to the executable.
      73             : - `ignore_unrecognized_command_line_options`: a `static constexpr bool` that
      74             :   defaults to `false`. If set to `true` then unrecognized command line options
      75             :   are ignored. Ignoring unrecognized options is generally only necessary for
      76             :   tests where arguments for the testing framework,
      77             :   [Catch](https://github.com/catchorg/Catch2/), are passed to the executable.
      78             : 
      79             : # Phases of an Execution {#dev_guide_parallelization_phases_of_execution}
      80             : 
      81             : Global synchronization points, where all cores wait for each other, are
      82             : undesirable for scalability reasons. However, they are sometimes inevitable for
      83             : algorithmic reasons. That is, in order to actually get a correct solution you
      84             : need to have a global synchronization. SpECTRE executables can have multiple
      85             : phases, where after each phase a global synchronization occurs. By global
      86             : synchronization we mean that no parallel components are executing or have more
      87             : tasks to execute: everything is waiting to be told what tasks to perform next.
      88             : 
      89             : Every executable must have at least two phases, `Initialization` and
      90             : `Exit`. The next phase is decided by the member function
      91             : Parallel::Main<Metavariables>::execute_next_phase.
      92             : Usually the next phase is determined
      93             : from the `default_phase_order` provided by the metavariables.  If more
      94             : complex decision making is desired, various components can send data
      95             : to Parallel::Main via the PhaseControl infrastructure.  This allows the next
      96             : Phase to be determined dynamically. Here is an example of a
      97             : `default_phase_order` member variable:
      98             : \snippet LinearSolverAlgorithmTestHelpers.hpp default_phase_order_array
      99             : 
     100             : In contrast, an evolution executable might have phases
     101             : `Initialization`, `SetInitialData`, `Evolve`, and `Exit`.
     102             : The first phase that is entered is always
     103             : `Initialization`. During the `Initialization` phase the
     104             : `Parallel::GlobalCache` is created, all (node)group components are created,
     105             : and empty array and singleton components are created.  Next, the function
     106             : `allocate_remaining_components_and_execute_initialization_phase` is called
     107             : which allocates singleton components and the elements of each array component,
     108             : and then starts the `Initialization` phase on all parallel components. Once all
     109             : parallel components' `Initialization` phase is complete, the next
     110             : phase is determined and the `execute_next_phase` function is called on
     111             : all the parallel components.
     112             : 
     113             : At the end of an execution the `Exit` phase has the executable wait to make sure
     114             : no parallel components are performing or need to perform any more tasks, and
     115             : then exits. An example where this approach is important is if we are done
     116             : evolving a system but still need to write data to disk. We do not want to exit
     117             : the simulation until all data has been written to disk, even though we've
     118             : reached the final time of the evolution.
     119             : 
     120             : \warning Currently dead-locks are treated as successful termination. In the
     121             : future checks against deadlocks will be performed before terminating.
     122             : 
     123             : # The Algorithm {#dev_guide_parallelization_core_algorithm}
     124             : 
     125             : Since most numerical algorithms repeat steps until some criterion such as the
     126             : final time or convergence is met, SpECTRE's parallel components are designed to
     127             : do such iterations for the user. An Algorithm executes an ordered list of
     128             : actions until one of the actions cannot be evaluated, typically because it is
     129             : waiting on data from elsewhere. When an algorithm can no longer evaluate actions
     130             : it passively waits by handing control back to Charm++. Once an algorithm
     131             : receives data, typically done by having another parallel component call the
     132             : `receive_data` function, the algorithm will try again to execute the next
     133             : action. If the algorithm is still waiting on more data then the algorithm will
     134             : again return control to Charm++ and passively wait for more data. This is
     135             : repeated until all required data is available. The actions that are iterated
     136             : over by the algorithm are called iterable actions and are described below.
     137             : Since the action list is phase dependent we refer to them generally as
     138             : phase-dependent action lists (PDALs, pronounced "pedals").
     139             : 
     140             : # Parallel Components {#dev_guide_parallelization_parallel_components}
     141             : 
     142             : Building off the introduction, a %Parallel Component is essentially a wrapper
     143             : around Charm++ chares that makes it easy for a user to add parallel objects into
     144             : their program. Charm++ chares can be confusing to work with which is why we wrap
     145             : them. Each parallel component runs its own \ref
     146             : dev_guide_parallelization_core_algorithm "Algorithm". Data can be sent from one
     147             : parallel component to another and the receiving parallel components' Algorithm
     148             : will be able to take that data and continue the program.
     149             : 
     150             : ## 1. Types of Parallel Components {#dev_guide_parallelization_component_types}
     151             : 
     152             : There are four types of %Parallel Components in SpECTRE:
     153             : 
     154             : 1. `Parallel::Algorithms::Singleton`s have one object in the entire execution of
     155             :    the program. They are implemented as single element Charm++ chare arrays.
     156             :    Charm++ does offer a distributed object called a singleton, however, we
     157             :    explicitly don't use this for various reasons (see
     158             :    Parallel::Algorithms::Singleton). Henceforth and throughout SpECTRE, a
     159             :    `singleton` will refer to Parallel::Algorithms::Singleton and *not* a Charm++
     160             :    singleton.
     161             : 2. `Parallel::Algorithms::Array`s hold zero or more elements, each of which is
     162             :    an object distributed to some core. An array can grow and shrink in size
     163             :    dynamically if need be and can also be bound to another array. A bound array
     164             :    has the same number of elements as the array it is bound to, and elements
     165             :    with the same ID are on the same core. See Charm++'s chare arrays for
     166             :    details.
     167             : 3. `Parallel::Algorithms::Group`s are arrays with one element per core which are
     168             :    not able to be moved around between cores. These are typically useful for
     169             :    gathering data from elements of a Parallel::Algorithms::Array on their core,
     170             :    and then processing or reducing the data further. See
     171             :    [Charm++'s](http://charm.cs.illinois.edu/help) group chares for details.
     172             : 4. `Parallel::Algorithms::Nodegroup`s are similar to groups except that there is
     173             :    one element per node. See [parallel component
     174             :    placement](#dev_guide_parallelization_component_placement) for the definition
     175             :    of a cores and nodes. We ensure that all entry method calls done through the
     176             :    Algorithm's `simple_action` and `receive_data` functions are threadsafe.
     177             :    User-controlled threading is possible by calling the entry method member
     178             :    function `threaded_action`, which is like `simple_action` except it passes a
     179             :    node lock to the `Action`'s apply function. Note that unlike
     180             :    `simple_action`s, multiple `threaded_action`s can be executing simultaneously
     181             :    on the same chare, but on different cores of the node.
     182             : 
     183             : \note Technically there is a fifth type of parallel component called a MainChare
     184             : which is placed and executed on the global zeroth core. However, there can only
     185             : be one of these in a executable, it is entirely different from a Singleton, and
     186             : it is not specifiable by the user so we omit it from the count here.
     187             : 
     188             : ## 2. Requirements {#dev_guide_parallelization_component_requirements}
     189             : 
     190             : Each %Parallel Component struct must have the following type aliases:
     191             : 1. `using chare_type` is set to one of the four \ref
     192             :    dev_guide_parallelization_component_types "types of Parallel Components".
     193             : 2. `using metavariables` is set to the Metavariables struct that stores the
     194             :    global metavariables. It is often easiest to have the %Parallel
     195             :    Component struct have a template parameter `Metavariables` that is the
     196             :    global metavariables struct. Examples of this technique are given below.
     197             : 3. `using phase_dependent_action_list` is set to a `tmpl::list` of
     198             :    `Parallel::PhaseActions<Phase, tmpl::list<Actions...>>`
     199             :    where each `PhaseAction` represents a PDAL that will be executed on
     200             :    the parallel component during the specified phase. The %Actions are
     201             :    executed in the order that they are given in the `tmpl::list`s of
     202             :    the PDALs, but the phases need not be run in linear order. However,
     203             :    `db::DataBox` types are constructed assuming the phases are
     204             :    performed from first in the `phase_dependent_action_list` to the
     205             :    last. Simple actions (described below) can be executed in any
     206             :    phase. If there are no iterable actions in a phase then a
     207             :    `PhaseAction` need not be specified for that phase. However, at
     208             :    least one `PhaseAction`, even if it is empty, must be specified.
     209             : 4. `using simple_tags_from_options` which is a `tmpl::list` of all the tags
     210             :    that will be inserted into the initial `db::DataBox` of each component.
     211             :    These tags are db::SimpleTag%s that have have a `using option_tags`
     212             :    type alias and a static function `create_from_options` (see the
     213             :    example below).  This list can usually be constructed from the
     214             :    initialization actions of the component (i.e. the list of actions
     215             :    in the `PhaseAction` list for the `Initialization` phase) using the
     216             :    helper function `Parallel::get_simple_tags_from_options` (see the
     217             :    examples of components below).  Each initialization action may
     218             :    specify a type alias `using simple_tags_from_options` which are a
     219             :    `tmpl::list` of tags that will be fetched from the db::DataBox by the
     220             :    action.
     221             : 5. `using const_global_cache_tags` is set to a `tmpl::list` of tags
     222             :    that are required by the `allocate_array` function of an array
     223             :    component, or simple actions called on the parallel component.
     224             :    These tags correspond to const items that are stored in the
     225             :    Parallel::GlobalCache (of which there is one copy per Charm++
     226             :    node).  The alias can be omitted if the list is empty.
     227             : 6. `using mutable_global_cache_tags` is set to a `tmpl::list` of tags
     228             :    that correspond to mutable items that are stored in the
     229             :    Parallel::GlobalCache (of which there is one copy per Charm++
     230             :    core).  The alias can be omitted if the list is empty.
     231             : 
     232             : \parblock
     233             : \note Array parallel components must also specify the type alias `using
     234             : array_index`, which is set to the type that indexes the %Parallel Component
     235             : Array. Charm++ allows arrays to be 1 through 6 dimensional or be indexed by a
     236             : custom type. The Charm++ provided indexes are wrapped as
     237             : `Parallel::ArrayIndex1D` through `Parallel::ArrayIndex6D`. When writing custom
     238             : array indices, the [Charm++ manual](http://charm.cs.illinois.edu/help) tells you
     239             : to write your own `CkArrayIndex`, but we have written a general implementation
     240             : that provides this functionality (see `Parallel::ArrayIndex`); all that you need
     241             : to provide is a plain-old-data
     242             : ([POD](http://en.cppreference.com/w/cpp/concept/PODType)) struct of the size of
     243             : at most 3 integers.
     244             : \endparblock
     245             : 
     246             : \parblock
     247             : \note Singletons use an `array_index` of type `int`, but users need not specify
     248             : this. It is already specified in the implementation of a singleton.
     249             : \endparblock
     250             : 
     251             : %Parallel array components have a static `allocate_array` function
     252             : that is used to construct the elements of the array. The
     253             : signature of the `allocate_array` functions must be:
     254             : \code
     255             : static void allocate_array(
     256             :     Parallel::CProxy_GlobalCache<metavariables>& global_cache,
     257             :     const tuples::tagged_tuple_from_typelist<simple_tags_from_options>&
     258             :     initialization_items,
     259             :     const std::unordered_set<size_t>& procs_to_ignore);
     260             : \endcode
     261             : The `allocate_array` function is called by the Main parallel component
     262             : when the execution starts and will typically insert elements into
     263             : array parallel components. If the `allocate_array` function depends
     264             : upon input options that are not in the GlobalCache, the array component must
     265             : include the appropriate tag in the `simple_tags_from_options` type alias.
     266             : This type alias is a `tmpl::list` of tags which
     267             : are db::SimpleTag%s that have have a `using option_tags` type alias
     268             : and a static function `create_from_options`. They only need to be explicitly
     269             : added to the list if no initialization action has added them to its
     270             : `simple_tags_from_options` type alias.  If you want to ignore specific
     271             : processors when placing array elements, you can pass in a
     272             : `std::unordered_set<size_t>` to `allocate_array` that contains all the
     273             : processors that shouldn't have array elements on them.
     274             : 
     275             : The `allocate_array` functions of different
     276             : array components are called in random order and so it is not safe to
     277             : have them depend on each other.
     278             : 
     279             : Each parallel component must also decide what to do in the different phases of
     280             : the execution. This is controlled by an `execute_next_phase` function with
     281             : signature:
     282             : \code
     283             : static void execute_next_phase(
     284             :     const Parallel::Phase next_phase,
     285             :     const Parallel::CProxy_GlobalCache<metavariables>& global_cache);
     286             : \endcode
     287             : Parallel::Main<Metavariables>::execute_next_phase`
     288             : determines the next phase, after
     289             : which the `execute_next_phase` function of each component gets called. The
     290             : `execute_next_phase` function determines what the parallel component should do
     291             : during the next phase. Typically the `execute_next_phase` function should just
     292             : call `start_phase(phase)` on the parallel component.

     314             : ## 4. Placement {#dev_guide_parallelization_component_placement}
     315             : 
     316             : The user has some control over where parallel components get placed on the
     317             : resources it is running on. Here is a figure that illustrates how one may place
     318             : parallel components.
     319             : 
     320             : \image html charm_node_structure.png "Parallel component placement."
     321             : 
     322             : In this example we are running on three (3) nodes that have four (4) cores each.
     323             : For all our executables, we reserve one core of each node purely for
     324             : communication purposes. Nothing else is run on this core. Because of this, what
     325             : Charm++ calls a node, doesn't correspond to a full node on a supercomputer. A
     326             : charm-node simply corresponds to a collection of cores on a physical node. In
     327             : our case, a charm-node is represented by the remaining cores on a node not used
     328             : for communication (i.e. the first charm-node corresponds to cores 1-3 on the
     329             : first physical node). Also the definition of a charm-core doesn't necessarily
     330             : have to correspond to an actual core (it could correspond to a hyperthreaded
     331             : virtual core), however, for our purposes, it does.
     332             : 
     333             : SpECTRE offers wrappers around Charm++ functions that will tell you the total
     334             : number of charm-nodes/cores in an executable and what charm-node/core a parallel
     335             : component is on. (In the following examples, the type `T` is an `int` or a
     336             : `size_t`)
     337             : 
     338             : - `Parallel::my_node<T>()` returns the charm-node that the parallel component is
     339             :   on. In the figure, `Sing. 4` would return `2`.
     340             : - `Parallel::my_proc<T>()` returns the charm-core that the parallel component is
     341             :   on. In the figure, `Sing. 4` would return `6` (*not* `9`).
     342             : - `Parallel::number_of_nodes<T>()` returns the total number of charm-nodes in an
     343             :   executable. The above figure would have `3` charm-nodes.
     344             : - `Parallel::number_of_procs<T>()` returns the total number of charm-cores in an
     345             :   executable. The above figure would have `9` charm-cores (*not* `12`).
     346             : 
     347             : \note For Charm++ SMP (shared memory parallelism) builds, a node corresponds to
     348             : a collection of cores on a physical node, and a core corresponds to a processor
     349             : on that physical node. However, for non-SMP builds, nodes and cores are
     350             : equivalent. All of our builds are done with Charm++ SMP so nodes and cores have
     351             : their usual definitions.
     352             : 
     353             : The placement of Groups and Nodegroups are determined by Charm++. This is
     354             : because a Group is on every charm-core and a Nodegroup is on every charm-node.
     355             : Even though Nodegroups are one per charm-node, the user can't choose which core
     356             : is used on the charm-node. They run on the next available charm-core on the
     357             : charm-node.
     358             : 
     359             : The Elements of an Array, however, can be placed on specific charm-cores. They
     360             : are inserted into the Array by using the Charm++ `insert` member function of the
     361             : CProxy for the Array. The `insert` function is documented in the Charm++ manual.
     362             : In the Array example in the
     363             : [Examples](#dev_guide_parallelization_component_examples) section, `array_proxy`
     364             : is a `CProxy` and so all the documentation for Charm++ array proxies applies.
     365             : SpECTRE always creates empty arrays with the constructor and requires users to
     366             : insert however many elements they want and on which charm-cores they want them
     367             : to be placed. Note that load balancing calls may result in array elements being
     368             : moved.
     369             : 
     370             : In a similar fashion, Singletons can also be placed on specific charm-cores.
     371             : This can be specified in the input file.
     372             : 
     373             : From an input file, there are two ways to specify where Array/Singleton parallel
     374             : components can be placed.
     375             : 
     376             : ```yaml
     377             : ResourceInfo:
     378             :   AvoidGlobalProc0: true
     379             :   Singletons:
     380             :     AhA:
     381             :       Proc: 12
     382             :       Exclusive: true
     383             :     AhB:
     384             :       Proc: Auto
     385             :       Exclusive: false
     386             : ```
     387             : 
     388             : First is the `AvoidGlobalProc0` option. This option will tell the program to not
     389             : put *any* Array Elements or Singletons on the global zeroth charm-core. This
     390             : core is sometimes used to write data to disk which is typically much slower
     391             : than the program execution. The second is the `Singletons:` option. You can
     392             : set the value to `Auto`, and then each singleton will have their proc be chosen
     393             : automatically and they won't be exclusive.  Otherwise, you must specify options
     394             : for each singleton as in the example above.  `AhA` is the `pretty_type::name()`
     395             : of a Singleton in the program and the user has a choice of which proc to place
     396             : the Singleton on (`Auto` will let the program decide) and whether to exclude
     397             : Array Elements or other Singletons from being put on this core. This is useful
     398             : in case the Singleton does some expensive computation that shouldn't be slowed
     399             : down by having lots of Array Elements on the same core. In the figure above,
     400             : `AvoidGlobalProc0` is true, and `Sing. 2` requested to be exclusively on core
     401             : `2`.
     402             : 
     403             : # Actions {#dev_guide_parallelization_actions}
     404             : 
     405             : %Actions are structs with a static `apply` method and come in five
     406             : variants: simple actions, iterable actions, reduction actions,
     407             : threaded actions, and local synchronous actions.
     408             : 
     409             : The signature of `apply` methods differs for the different types of
     410             : actions, but all types have the same general form.  Actions receive a
     411             : `db::DataBox`, the Parallel::GlobalCache, and their element's index
     412             : and parallel component, as well as arguments specific to the action
     413             : type.
     414             : 
     415             : The `db::DataBox` should be thought of as the member data of the parallel
     416             : component while the actions are the member functions. The combination of a
     417             : `db::DataBox` and actions allows building up classes with arbitrary member data
     418             : and methods using template parameters and invocation of actions. This approach
     419             : allows us to eliminate the need for users to work with Charm++'s interface
     420             : files, which can be error prone and difficult to use.
     421             : 
     422             : The Parallel::GlobalCache is passed to each action so that the
     423             : action has access to global data and is able to invoke actions on
     424             : other parallel components. The `ParallelComponent` template parameter
     425             : is the tag of the parallel component that invoked the action. A proxy
     426             : to the calling parallel component can then be retrieved from the
     427             : Parallel::GlobalCache. The remote entry method invocations are
     428             : slightly different for different types of actions, so they will be
     429             : discussed below. However, one thing that is disallowed for all actions
     430             : is calling an action locally from within an action on the same

     468             : ## 1. Simple Actions {#dev_guide_parallelization_simple_actions}
     469             : 
     470             : Simple actions can be thought of as member functions of remote objects
     471             : (chares/parallel components).  They are the direct analog of entry
     472             : methods in Charm++ except that the member data is stored in the
     473             : `db::DataBox` that is passed in as the first argument. A simple action
     474             : must return void but can use `db::mutate` to change values of items in
     475             : the `db::DataBox` if the `db::DataBox` is taken as a non-const
     476             : reference. In some cases you will need specific items to be in the
     477             : `db::DataBox` otherwise the action won't compile. To restrict which
     478             : `db::DataBox`es can be passed you should use `Requires` in the
     479             : action's `apply` function template parameter list. For example,

     492             : ## 2. Iterable Actions {#dev_guide_parallelization_iterable_actions}
     493             : 
     494             : Iterable actions make up the algorithms described by the PDALs.  These
     495             : actions are executed one after the other until one of them cannot be
     496             : evaluated.  Their `apply` methods signature is
     497             : \snippet Test_AlgorithmCore.cpp apply_iterative
     498             : The `ActionList` type is the `tmpl::list` of iterable actions in the
     499             : current phase.  That is, it is equal to the `action_list` type alias
     500             : in the current PDAL.  The `inboxes` is a collection of the tags
     501             : specified as `tmpl::list`s in the iterable actions' member type
     502             : aliases `inbox_tags`.  This collection represents data received from
     503             : other chares using the `receive_data` function.
     504             : 
     505             : Iterable actions can request that the algorithm be paused or halted
     506             : for the current phase, and control which action in the current PDAL
     507             : will be executed next. This is all done via the return value from the
     508             : `apply` function.  The `apply` function for iterable actions must
     509             : return a Parallel::iterable_action_return_t which is a
     510             : `std::tuple<Parallel::AlgorithmExecution, std::optional<size_t>>`. The
     511             : first element of the tuple controls how algorithm execution continues.
     512             : See the documentation of `Parallel::AlgorithmExecution` for the
     513             : meanings of different values of that enum.  The second element of the
     514             : tuple is usually set to `std::nullopt` in order to continue iterating
     515             : through the algorithm, but can be used to jump to a different action
     516             : in the current PDAL. Most iterable actions will simply return
     517             : 
     518             : \snippet Test_AlgorithmParallel.cpp iterable_action_return_continue_next_action
     519             : 
     520             : An action that pauses the algorithm will return
     521             : 
     522             : \snippet Test_AlgorithmParallel.cpp return_with_termination
     523             : 
     524             : After an algorithm has been paused it can be restarted by passing
     525             : `false` to the `set_terminate` method or by calling `receive_data(...,
     526             : true)`. Since the order in which messages are received is undefined in
     527             : most cases the `receive_data(..., true)` call should be used to
     528             : restart the algorithm.
     529             : 
     530             : The return value `Parallel::AlgorithmExecution::Retry` deserves
     531             : special mention.  It is intended for use by actions that use data
     532             : supplied by other parallel objects to indicate that they have not
     533             : received all of the data they require.  The algorithm will stop until
     534             : an operation that could supply data has occurred and then the action
     535             : will be retried.  An example of waiting because of missing data is
     536             : 
     537             : \snippet Test_AlgorithmCore.cpp retry_example
     538             : 
     539             : In order to jump to a specific action, the metafunction
     540             : `tmpl::index_of<list, element>` can be used to get an
     541             : `tmpl::integral_constant` with the value of the index of the element
     542             : `element` in the typelist `list`. For example,
     543             : 
     544             : \snippet Test_AlgorithmCore.cpp out_of_order_action
     545             : 
     546             : The metafunction call `tmpl::index_of<ActionList,
     547             : iterate_increment_int0>::%value` returns a `size_t` whose value is
     548             : that of the action `iterate_increment_int0` in the PDAL.  The indexing
     549             : of actions in the PDAL starts at `0`.
     550             : 
     551             : Iterable actions are invoked as part of the algorithm and so the only way
     552             : to request they be invoked is by having the algorithm run on the parallel
     553             : component. The algorithm can be explicitly evaluated in a new phase by calling
     554             : `start_phase(Phase::TheCurrentPhase)`:
     555             : 
     556             : \snippet Test_AlgorithmCore.cpp start_phase
     557             : 
     558             : Alternatively, to evaluate the algorithm without changing phases the
     559             : `perform_algorithm()` method can be used.
     560             : 
     561             : By passing `true` to `perform_algorithm` the algorithm will be restarted if it
     562             : was paused.
     563             : 
     564             : The algorithm is also evaluated by calling the `receive_data` function, either
     565             : on an entire array or singleton (this does a broadcast), or an on individual
     566             : element of the array. Here is an example of a broadcast call:
     567             : 
     568             : \snippet Test_AlgorithmParallel.cpp broadcast_to_group
     569             : 
     570             : and of calling individual elements:

     574             : The `receive_data` function always takes a `ReceiveTag`, which is set
     575             : in the actions' `inbox_tags` type aliases.  The `inbox_tags` must have
     576             : two member type aliases, a `temporal_id` which is used to identify
     577             : when the data was sent, and a `type` which is the type of the data to
     578             : be stored in the `inboxes`.  The types are typically a
     579             : `std::unordered_map<temporal_id, DATA>`.  In the discussed scenario of
     580             : waiting for neighboring elements to send their data the `DATA` type
     581             : would be a `std::unordered_map<TheElementId, DataSent>`.  Inbox tags
     582             : must also specify a `static void insert_into_inbox()` function.  For
     583             : example,

     586             :

     588             : function, a data structure with a `push_back` function, or copy/move assignment
     589             : that is used to insert the received data, inserters are available in
     590             : `Parallel::InboxInserters`. For example, there is
     591             : `Parallel::InboxInserters::Map` for `map` data structures. The inbox tag can

     600             :

     606             : It is the responsibility of the iterable action to remove data from the inboxes
     607             : that will no longer be needed.
     608             : 
     609             : Normally when remote functions are invoked they go through the Charm++ runtime
     610             : system, which adds some overhead. The `receive_data` function tries to elide
     611             : the call to the Charm++ RTS for calls into array components. Charm++ refers to
     612             : these types of remote calls as "inline entry methods". With the Charm++ method
     613             : of eliding the RTS, the code becomes susceptible to stack overflows because
     614             : of infinite recursion. The `receive_data` function is limited to at most 64 RTS
     615             : elided calls, though in practice reaching this limit is rare. When the limit is
     616             : reached the remote method invocation is done through the RTS instead of being
     617             : elided.

     619             : ## 3. Reduction Actions {#dev_guide_parallelization_reduction_actions}
     620             : 
     621             : Reduction actions are the targets of reducing data over an array. For
     622             : example, you may want to know the sum of a `int` from every element in
     623             : the array. You can do this as follows:
     624             : 
     625             : \snippet Test_AlgorithmReduction.cpp contribute_to_reduction_example
     626             : 
     627             : This reduces over the parallel component
     628             : `ArrayParallelComponent<Metavariables>`, reduces to the parallel component
     629             : `SingletonParallelComponent<Metavariables>`, and calls the action
     630             : `ProcessReducedSumOfInts` after the reduction has been performed. The reduction
     631             : action is:
     632             : 
     633             : \snippet Test_AlgorithmReduction.cpp reduce_sum_int_action
     634             : 
     635             : As you can see, the last argument to the `apply` function is of type `int`, and
     636             : is the reduced value.
     637             : 
     638             : You can also broadcast the result back to an array, even yourself. For example,
     639             : 
     640             : \snippet Test_AlgorithmReduction.cpp contribute_to_broadcast_reduction
     641             : 
     642             : It is often necessary to reduce custom data types, such as `std::vector` or
     643             : `std::unordered_map`. Charm++ supports such custom reductions, and so does our
     644             : layer on top of Charm++.
     645             : Custom reductions require one additional step to calling
     646             : `contribute_to_reduction`, which is writing a reduction function to reduce the
     647             : custom data. We provide a generic type that can be used in custom reductions,
     648             : `Parallel::ReductionData`, which takes a series of `Parallel::ReductionDatum` as
     649             : template parameters and `ReductionDatum::value_type`s as the arguments to the
     650             : constructor. Each `ReductionDatum` takes up to four template parameters (two
     651             : are required). The first is the type of data to reduce, and the second is a
     652             : binary invokable that is called at each step of the reduction to combine two
     653             : messages. The last two template parameters are used after the reduction has
     654             : completed. The third parameter is an n-ary invokable that is called once the
     655             : reduction is complete, whose first argument is the result of the reduction. The
     656             : additional arguments can be any `ReductionDatum::value_type` in the
     657             : `ReductionData` that are before the current one. The fourth template parameter
     658             : of `ReductionDatum` is used to specify which data should be passed. It is a
     659             : `std::index_sequence` indexing into the `ReductionData`.
     660             : 
     661             : \warning
     662             : All elements of the array must call the same reductions in the same order. It is
     663             : *defined* behavior to do multiple reductions at once as long as all contribute
     664             : calls on all array elements occurred in the same order. It is **undefined**
     665             : behavior if the contribute calls are made in different orders on different array
     666             : elements.

     668             : ## 4. Threaded Actions {#dev_guide_parallelization_threaded_actions}
     669             : 
     670             : Threaded actions are similar to simple actions, with the difference
     671             : being that multiple threaded actions may be running on the same chare
     672             : at the same time (potentially in parallel with one simple or reduction
     673             : action).  The `apply` function for a threaded actions has the same
     674             : signature as that for a simple action, except that it also receives a
     675             : `NodeLock` intended to control access to the chare's `db::DataBox`.
     676             : All access to the `db::DataBox`, including read-only access, must
     677             : occur while the action owns this lock.  (Simple and reduction actions
     678             : implicitly hold the lock for their entire execution.)
     679             : 
     680             : \snippet Test_AlgorithmNodelock.cpp threaded_action_example
     681             : 
     682             : Threaded actions can only be run on nodegroup chares.
     683             : 
     684             : ## 5. Local Synchronous Actions {#dev_guide_parallelization_local_synchronous_actions}
     685             : 
     686             : There is limited ability to retrieve data held by another parallel component via
     687             : a direct synchronous call. Unlike the above actions, the invocation of a
     688             : synchronous action is precisely a call to a member function of another parallel
     689             : component; therefore, these invocations will run to completion, and return their
     690             : result before the calling code proceeds in execution.
     691             : 
     692             : Aside from being synchronous and being able to return data, local
     693             : synchronous actions behave the same as threaded actions, except that
     694             : they will only run on the chare of a nodegroup that is on the local
     695             : node.
     696             : 
     697             : Local synchronous actions' `apply` functions follow a signature motivated by
     698             : threaded actions, but take fewer arguments.  This may be a bug.
     699             : 
     700             : Local synchronous actions must specify their return type in a
     701             : `return_type` type alias. This is to help simplify the logic with the
     702             : variant `db::DataBox` held by the parallel component.

     712             : # Mutable items in the GlobalCache {#dev_guide_parallelization_mutable_global_cache}
     713             : 
     714             : Most items in the GlobalCache are constant, and are specified
     715             : by type aliases called `const_global_cache_tags` as
     716             : described above. However, the GlobalCache can also store mutable
     717             : items. Because of asynchronous execution, care must be taken when
     718             : mutating items in the GlobalCache, as described below.
     719             : 
     720             : A mutable item can be of any type, as long as that type is something
     721             : that can be checked for whether it is "up-to-date".  Here "up-to-date"
     722             : means that the item can be safely used (even read-only) without
     723             : needing to be mutated first. For example, a mutable item might be a
     724             : function of time that knows the range of times for which it is valid;
     725             : the mutable item is then deemed up-to-date if it will be called for a
     726             : time within its range of validity, and it is deemed not up-to-date if
     727             : it will be called for a time outside its range of validity.  Thus the
     728             : up-to-date status of a mutable item is determined by both the state of
     729             : the item itself and by the code that wishes to use that item.
     730             : 
     731             : ## 1. Specification of mutable GlobalCache items
     732             : 
     733             : Mutable GlobalCache items are specified by a
     734             : type alias `mutable_global_cache_tags`, which is treated the same way
     735             : as `const_global_cache_tags` for const items.
     736             : 
     737             : ## 2. Use of mutable GlobalCache items
     738             : 
     739             : ### 1. Checking if the item is up-to-date
     740             : 
     741             : Because execution is asynchronous, any code that uses a mutable item
     742             : in the GlobalCache must first check whether that item is up-to-date.
     743             : The information about whether an item is up-to-date is assumed to be
     744             : stored in the item itself.  For example, a mutable object stored in
     745             : the GlobalCache might have type `std::map<temporal_id,T>` (for some
     746             : type `T`), and then any code that uses the stored object can check
     747             : whether an entry exists for a particular `temporal_id`.  To avoid
     748             : race conditions, it is
     749             : important that up-to-date checks are based on something that is
     750             : independent of the order of mutation (like a `temporal_id`, and not
     751             : like checking the size of a vector).
     752             : 
     753             : To check an item, use the function
     754             : `Parallel::mutable_cache_item_is_ready`, which returns a bool
     755             : indicating whether the item is up-to-date.  If the item is up-to-date,
     756             : then it can be used.  `Parallel::mutable_cache_item_is_ready` takes a
     757             : lambda as an argument.  This lambda is passed a single argument: a
     758             : const reference to the item being retrieved.  The lambda should
     759             : determine whether the item is up-to-date. If so, it should return a
     760             : default_constructed `std::unique_ptr<Parallel::Callback>`; if not, it should
     761             : return a `std::unique_ptr<Parallel::Callback>` to a callback function that will
     762             : be called on the next `Parallel::mutate` of that item. The callback
     763             : will typically check again if the item is up-to-date and if so will
     764             : execute some code that gets the item via `Parallel::get`.
     765             : 
     766             : For the case of iterable actions,
     767             : `Parallel::mutable_cache_item_is_ready` typically uses the callback
     768             : `Parallel::PerformAlgorithmCallback`.  In the example below, the
     769             : vector is considered up-to-date if it is non-empty. If the vector is
     770             : not up-to-date, then when it becomes up-to-date the callback function
     771             : will be invoked; in this case the callback function re-runs
     772             : `perform_algorithm()`, which will call the same action again.
     773             : 
     774             : \snippet Test_AlgorithmGlobalCache.cpp check_mutable_cache_item_is_ready
     775             : 
     776             : Note that `Parallel::mutable_cache_item_is_ready` is called on a local
     777             : core and does no parallel communication.
     778             : 
     779             : ### 2. Retrieving the item
     780             : 
     781             : The item is retrieved using `Parallel::get` just like for constant items.
     782             : For example, to retrieve the item `Tags::VectorOfDoubles`:
     783             : \snippet Test_AlgorithmGlobalCache.cpp retrieve_mutable_cache_item
     784             : 
     785             : Note that `Parallel::get` is called on a local core and does no
     786             : parallel communication.
     787             : 
     788             : Whereas we support getting *non-mutable* items in the GlobalCache from
     789             : a DataBox via `db::get`, we intentionally do not support
     790             : `db::get` of *mutable* items in the GlobalCache from a DataBox.
     791             : The reason is that mutable
     792             : items should be retrieved only after a `Parallel::mutable_cache_item_is_ready`
     793             : check, and being able to retrieve a mutable item from a DataBox makes it
     794             : difficult to enforce that check, especially when automatically-executing
     795             : compute items are considered.
     796             : 
     797             : ## 3. Modifying a mutable GlobalCache item

     799             : To modify a mutable item, pass `Parallel::mutate` two template
     800             : parameters: the tag to mutate, and a struct with an `apply` function
     801             : that does the mutating. `Parallel::mutate` takes two arguments:
     802             : a reference to the local GlobalCache, and a tuple that is passed into the
     803             : mutator function.  For the following example,
     804             :
