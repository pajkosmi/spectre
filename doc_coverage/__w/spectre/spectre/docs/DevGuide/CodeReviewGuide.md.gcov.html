      16             : Here we summarize what we view as the more important portions of the guides.
      17             : 
      18             : Stylistic Items:
      19             : 
      20             : * Adhere to [Google style](https://google.github.io/styleguide/cppguide.html).
      21             :   [Can use `clang-format -style=google`]
      22             :   (http://clang.llvm.org/docs/ClangFormat.html).
      23             : * CamelCase: class names template parameters, file names, and directory names.
      24             : * snake_case: function, variable, metafunction and metavariable names.
      25             : * SCREAMING_SNAKE_CASE: macros.
      26             : * Functions or classes only used internally in a library should be in a
      27             :   namespace named `LibraryOrFile_detail`. For example, `databox_detail`,
      28             :   `Tensor_detail` or `ConstantExpressions_detail`.
      29             : * Name unused function parameters `/*parameter_name*/` or `/*meta*/` for TMP
      30             :   cases
      31             : * Type aliases that wrap type traits have a trailing `_t` following the STL
      32             : * Private member variables have a [trailing underscore]
      33             :   (https://google.github.io/styleguide/cppguide.html#Variable_Names).
      34             : * Do not use
      35             :   [Hungarian notation](https://en.wikipedia.org/wiki/Hungarian_notation),
      36             :   e.g. `double* pd_blah` is bad
      37             : * Header order:
      38             :   1. (If a test:) `Framework/TestingFramework.hpp`, followed by one blank line
      39             :   2. (If a cpp file with a corresponding hpp file:) hpp corresponding to cpp,
      40             :      followed by one blank line
      41             :   3. STL and externals (in alphabetical order)
      42             :   4. Blank line
      43             :   5. SpECTRE includes (in alphabetical order)
      44             : * Template definitions in header files are separated from the declaration of
      45             :   the class by the following line, which contains exactly 64 equal signs
      46             :   ``` cpp
      47             :   // ================================================================
      48             :   ```
      49             : 
      50             : * File lists in CMake are alphabetical.
      51             : * No blank lines surrounding Doxygen group comments
      52             :   (<code>// \@{</code> and <code>// \@}</code>).
      53             : * Use the [alternative tokens]
      54             :   (http://en.cppreference.com/w/cpp/language/operator_alternative)
      55             :   `or`, `and`, and `not` instead of `||`, `&&`, and `!`.
      56             : * Use C-style Doxygen comments (`/*! ... */`) when using multi-line math,
      57             :   otherwise C-style and C++ style comments are accepted.
      58             : * Use the `align` environment instead of `eqnarray`. See the
      59             :   [texfaq](https://texfaq.org/FAQ-eqnarray) for an explanation as to why.
      60             : * Multi-line equations must have a blank Doxygen line before and after the
      61             :   equation.
      62             : * When addressing requests on a PR, the commit message must start with
      63             :   `fixup` followed by a descriptive message.
      64             : * All python code must comply with the [black](https://github.com/psf/black)
      65             :   formatting specified by the `pyproject.toml` file.
      66             : * Imports in Python code must be sorted to comply to the
      67             :   [isort](https://github.com/PyCQA/isort) formatting specified in the
      68             :   `pyproject.toml` file.
      69             : 
      70             : Code Quality Items:
      71             : 
      72             : * All code passes Clang and CppCheck static analyzers. For help
      73             :   with these tools see \ref static_analysis_tools "here".
      74             : * Almost always `auto`, except with expression templates, i.e. `DataVector`
      75             : * All loops and if statements use braces.
      76             : * Order of declaration in classes is `public` before `private` and member
      77             :   functions before data. See
      78             :   [Google](https://google.github.io/styleguide/cppguide.html#Declaration_Order).
      79             : * Prefer return by value over pass-by-mutable-reference except when mutable
      80             :   reference provides superior performance (in practice if you need a mutable
      81             :   reference use `const gsl::not_null<Type*>` instead of `Type&`). The mutable
      82             :   references must be the first arguments passed to the function.
      83             : * All commits for performance changes provide quantitative evidence and the
      84             :   tests used to obtain said evidence.
      85             : * Never include `<iostream>`, use `Parallel::printf` inside
      86             :   `Parallel/Printf.hpp` instead, which is safe to use in parallel.
      87             : * When using charm++ nodelocks include `<converse.h>` instead of `<lrtslock.h>`.
      88             : * Do not add anything to [the `std` namespace]
      89             :   (http://en.cppreference.com/w/cpp/language/extending_std).
      90             : * Virtual functions are explicitly overridden using the `override` keyword.
      91             : * `#%pragma once` is to be used for header guards
      92             : * Prefer range-based for loops
      93             : * Use `size_t` for positive integers rather than `int`, specifically when
      94             :   looping over containers. This is in compliance with what the STL uses.
      95             : * Error messages should be helpful. An example of a bad error message is "Size
      96             :   mismatch". Instead this message could read "The number of grid points in the
      97             :   matrix 'F' is not the same as the number of grid points in the determinant.",
      98             :   along with the runtime values of the mentioned quantities if applicable.
      99             : * Mark immutable objects as `const`
     100             : * Make classes serializable by writing a `pup` function
     101             : * If a class stores an object passed into a constructor the object should
     102             :   be taken by-value and `std::move`d into the member variable.
     103             : * Definitions of function and class templates should be in `.cpp` files with
     104             :   explicit instantiations whenever possible. The macro
     105             :   `GENERATE_EXPLICIT_INSTANTIATIONS` is useful for generating many
     106             :   explicit instantiations.
     107             : * Functions should not be marked `noexcept` unless it is part of the
     108             :   signature of an overriden function (e.g. the what() function of an exception
     109             :   derived from std::exception).
     110             : * Variable names in macros must avoid name collisions, e.g. inside the
     111             :   `PARSE_ERROR` macro you would write
     112             :   `double variable_name_avoid_name_collisions_PARSE_ERROR = 0.0;`
     113             : * Avoid macros if possible. Prefer `constexpr` functions, constexpr
     114             :   variables, and/or template metaprogramming
     115             : * Explicitly specify `double`s, e.g. `sqrt(2.)` or `sqrt(2.0)` instead of
     116             :   `sqrt(2)`.
     117             : * When the index of a `Tensor` is known at compile time, use
     118             :   `get<a, b>(tensor)` instead of the member `get` function.
     119             : * All necessary header files must be included. In header files, prefer
     120             :   forward declarations if class definitions aren't necessary.
     121             : * Explicitly make numbers floating point, e.g. `2.` or `2.0` over `2`.
     122             : * Avoid `mutable` member variables. If you think you really need these, discuss
     123             :   with at least 2 core developers. In a parallel environment `mutable` variables
     124             :   lead to race conditions since multiple threads can mutate a single object by
     125             :   calling a `const` member function.
