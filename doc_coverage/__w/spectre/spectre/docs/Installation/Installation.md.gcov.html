      17             : ### Install a specific version with Spack
      18             : 
      19             : You can install SpECTRE with the [Spack](https://github.com/spack/spack) package
      20             : manager:
      21             : 
      22             : ```sh
      23             : git clone https://github.com/spack/spack
      24             : source ./spack/share/spack/setup-env.sh
      25             : spack compiler find
      26             : spack external find
      27             : spack install spectre executables=ExportCoordinates3D \
      28             :   ^charmpp backend=multicore
      29             : ```
      30             : 
      31             : You probably want to customize your installation, e.g., to select a particular
      32             : version of SpECTRE, the executables you want to install, additional options such
      33             : as Python bindings, or the Charm++ backend. You can display all possible options
      34             : with:
      35             : 
      36             : ```sh
      37             : spack info spectre  # or charmpp, etc.
      38             : ```
      39             : 
      40             : Refer to the [Spack documentation](https://spack.readthedocs.io/en/latest/) for
      41             : more information.

     139             : ## Clone the SpECTRE repository
     140             : 
     141             : First, clone the [SpECTRE repository](https://github.com/sxs-collaboration/spectre)
     142             : to a directory of your choice. In the following we will refer to it as
     143             : SPECTRE_ROOT. You may `git clone` from GitHub, in which case SPECTRE_ROOT will
     144             : be `<your_current_directory>/spectre`. That is, inside SPECTRE_ROOT are `docs`,
     145             : `src`, `support`, `tests` etc. You can also download the source and extract them
     146             : to your desired working directory, making sure not to leave out hidden files
     147             : when you `cp` or `mv` the source files.

     149             : ## Using Docker to obtain a SpECTRE environment {#docker_install}
     150             : 
     151             : A [Docker](https://www.docker.com/) image is available from
     152             : [DockerHub](https://hub.docker.com/r/sxscollaboration/spectre/) and can
     153             : be used to build SpECTRE on a personal machine.
     154             : 
     155             : **Note**: The Docker image or the Singularity image (see below) are
     156             : the recommended ways of using SpECTRE on a personal
     157             : Linux machine. Because of the wide variety of operating systems available today
     158             : it is not possible for us to support all configurations. However, using Spack
     159             : as outlined below is a supported alternative to Docker or Singularity images.
     160             : 
     161             : **Note**: If you have SELinux active
     162             : on your system you must figure out how to enable sharing files with the host
     163             : OS. If you receive errors that you do not have permission to access a shared
     164             : directory it is likely that your system has SELinux enabled. One option is to
     165             : disable SELinux at the expense of reducing the security of your system.
     166             : 
     167             : To build with the Docker image:
     168             : 
     169             : 1. Install [Docker-Desktop](https://docs.docker.com/get-docker/). For Linux, if
     170             :    you want to be able to run the following steps without `sudo`, follow the
     171             :    [post-installation-guide](https://docs.docker.com/engine/install/linux-postinstall/)
     172             :    to add a non-root user.
     173             : 
     174             : 2. Retrieve the Docker image (you may need `sudo` in front of this command)
     175             :    ```
     176             :    docker pull sxscollaboration/spectre:dev
     177             :    ```
     178             : 3. Start the Docker container (you may need `sudo`)
     179             :    ```
     180             :    docker run -v $SPECTRE_ROOT/:$SPECTRE_ROOT/ --name spectre_dev \
     181             :               -i -t sxscollaboration/spectre:dev /bin/bash
     182             :    ```
     183             :    - `-v $SPECTRE_ROOT/:$SPECTRE_ROOT/` binds the directory `$SPECTRE_ROOT`
     184             :    (which is an environment variable you must set up or just use the actual
     185             :    path) outside the container to `$SPECTRE_ROOT` inside the container. In this
     186             :    way, files in the `$SPECTRE_ROOT` on your host system (outside the container)
     187             :    become accessible within the container through the directory SPECTRE_ROOT
     188             :    inside the container. If you wonder why the same SPECTRE_ROOT needs to be
     189             :    used for both inside and outside the container, which is why `$SPECTRE_ROOT`
     190             :    is repeated in the command above with separated by a colon, please see one of
     191             :    the notes below regarding `-v` flag.
     192             :    - The `--name spectre_dev` is optional. If you don't name your container,
     193             :    docker will generate an arbitrary name.
     194             :    - On macOS you can significantly increase the performance of file system
     195             :    operations by appending the flag `:delegated` to `-v`, e.g.
     196             :    `-v $SPECTRE_ROOT/:$SPECTRE_ROOT/:delegated` (see
     197             :    https://docs.docker.com/docker-for-mac/osxfs-caching/).
     198             :    - The `-i` flag is for interactive mode, which will drop you into the
     199             :    container.
     200             :    - It can be useful to expose a port to the host so you can run servers such
     201             :    as [Jupyter](https://jupyter.org/index.html) for accessing the Python
     202             :    bindings (see \ref spectre_using_python) or a Python web server to view the
     203             :    documentation. To do so, append the `-p` option, e.g. `-p 8000:8000`.
     204             : 
     205             :    You will end up in a bash shell in the docker container,
     206             :    as root (you need to be root).
     207             :    Within the container, the files in `$SPECTRE_ROOT` are available and Charm++
     208             :    is installed in `/work/charm_7_0_0`. For the following steps, stay inside the
     209             :    docker container as root.
     210             : 4. Proceed with [building SpECTRE](#building-spectre).
     211             : 
     212             : **Notes:**
     213             :   * Everything in your build directory is owned by root, and is
     214             :     accessible only within the container.
     215             :   * You should edit source files in SPECTRE_ROOT in a separate terminal
     216             :     outside the container, and use the container only for compiling and
     217             :     running the code.
     218             :   * If you exit the container (e.g. ctrl-d),
     219             :     your compilation directories are still saved, as are any other changes to
     220             :     the container that you have made.
     221             :     To restart the container, try the following commands
     222             :     (you may need `sudo`):
     223             :     1. `docker ps -a`,
     224             :       to list all containers with their CONTAINER_IDs and CONTAINER_NAMEs,
     225             :     2. `docker start -i CONTAINER_NAME` or `docker start -i CONTAINER_ID`,
     226             :       to restart your container (above, the CONTAINER_NAME was spectre_dev).
     227             :   * When the Docker container gets updated, you can stop it with
     228             :     `docker stop CONTAINER_NAME`, remove it with `docker rm CONTAINER_NAME`
     229             :     and then start at step 2 above to run it again.
     230             :   * You can run more than one shell in the same container, for instance
     231             :     one shell for compiling with gcc and another for compiling
     232             :     with clang.
     233             :     To add a new shell, run `docker exec -it CONTAINER_NAME /bin/bash`
     234             :     (or `docker exec -it CONTAINER_ID /bin/bash`) from
     235             :     a terminal outside the container.
     236             :   * In step 4 above, technically docker allows you to say
     237             :     `-v $SPECTRE_ROOT/:/my/new/path` to map `$SPECTRE_ROOT` outside the
     238             :     container to any path you want inside the container, but **do not do this**.
     239             :     Compiling inside the container sets up git hooks in SPECTRE_ROOT that
     240             :     contain hardcoded pathnames to SPECTRE_ROOT *as seen from inside the
     241             :     container*. So if your source paths inside and outside the container are
     242             :     different, commands like `git commit` run *from outside the container* will
     243             :     die with `No such file or directory`.
     244             :   * If you want to use Docker within VSCode, take a look at our
     245             :     [quick start guide](../DevGuide/QuickStartDockerVSCode.md) for using Docker
     246             :     with VSCode.

     248             : ## Using Singularity to obtain a SpECTRE environment
     249             : 
     250             : [Singularity](https://sylabs.io) is a container alternative
     251             : to Docker with better security and nicer integration.
     252             : 
     253             : To build SpECTRE with Singularity you must:
     254             : 
     255             : 1. Build [Singularity](https://sylabs.io) and add it to your
     256             :    `$PATH`
     257             : 2. `cd` to the directory where you want to store the SpECTRE Singularity image,
     258             :    source, and build directories, let's call it WORKDIR. The WORKDIR must be
     259             :    somewhere in your home directory. If this does not work for you, follow the
     260             :    Singularity instructions on setting up additional [bind
     261             :    points](https://sylabs.io/guides/3.7/user-guide/bind_paths_and_mounts.html)
     262             :    (version 3.7. For other versions, see the [docs](https://sylabs.io/docs/)).
     263             :    Once inside the WORKDIR, clone SpECTRE into `WORKDIR/SPECTRE_ROOT`.
     264             : 3. Run `sudo singularity build spectre.img
     265             :    docker://sxscollaboration/spectre:dev`.
     266             :    You can also use spectre:ci instead of spectre:dev if you want more
     267             :    compilers installed.
     268             : 
     269             :    If you get the error message that `makesquashfs` did not have enough space to
     270             :    create the image you need to set a different `SINGULARITY_TMPDIR`. This can
     271             :    be done by running: `sudo SINGULARITY_TMPDIR=/path/to/new/tmp singularity
     272             :    build spectre.img docker://sxscollaboration/spectre:dev`. Normally
     273             :    `SINGULARITY_TMPDIR` is `/tmp`, but building the image will temporarily need
     274             :    almost 8GB of space.
     275             : 
     276             :    You can control where Singularity stores the downloaded image files from
     277             :    DockerHub by specifying the `SINGULARITY_CACHEDIR` environment variable. The
     278             :    default is `$HOME/.singularity/`. Note that `$HOME` is `/root` when running
     279             :    using `sudo`.
     280             : 4. To start the container run `singularity shell spectre.img` and you
     281             :    will be dropped into a bash shell.
     282             : 5. Proceed with [building SpECTRE](#building-spectre).
     283             : 
     284             : **Notes:**
     285             : - You should edit source files in SPECTRE_ROOT in a separate terminal
     286             :   outside the container, and use the container only for compiling and running
     287             :   the code.
     288             : - If you don't have the same Python version in your environment outside the
     289             :   container as the version inside the container, this will create problems
     290             :   with git hooks. The Singularity container uses python3.8 by default. Thus, it
     291             :   is up to the user to ensure that they are using the same Python version inside
     292             :   and outside the container. To use a different Python version in the container
     293             :   add `-D Python_EXECUTABLE=/path/to/python` to the cmake command where
     294             :   `/path/to/python` is usually `/usr/bin/pythonX` and `X` is the version you
     295             :   want.
     296             : - Unlike Docker, Singularity does not keep the state between runs. However, it
     297             :   shares the home directory with the host OS so you should do all your work
     298             :   somewhere in your home directory.
     299             : - To run more than one container just do `singularity shell spectre.img` in
     300             :   another terminal.
     301             : - Since the data you modify lives on the host OS there is no need to worry about
     302             :   losing any data, needing to clean up old containers, or sharing data between
     303             :   containers and the host.

     305             : ## Using Spack to set up a SpECTRE environment
     306             : 
     307             : SpECTRE's dependencies can be installed with

     310             : cloning it into `SPACK_DIR` (a directory of your choice). Then, enable Spack's
     311             : shell support with `source SPACK_DIR/share/spack/setup-env.sh`. Consider adding
     312             : this line to your `.bash_profile`, `.bashrc`, or similar. Refer to [Spack's
     313             : getting started guide](https://spack.readthedocs.io/en/latest/getting_started.html)
     314             : for more information.
     315             : 
     316             : Once you have Spack installed, one way to install the SpECTRE dependencies is
     317             : with a [Spack environment](https://spack.readthedocs.io/en/latest/environments.html):
     318             : 
     319             : \include support/DevEnvironments/spack.yaml
     320             : 
     321             : You can also install the Spack packages listed in the environment file above
     322             : with a plain `spack install` if you prefer.
     323             : 
     324             : **Notes:**
     325             : - Spack allows very flexible configurations and we recommended you read the
     326             :   [documentation](https://spack.readthedocs.io) if you require features such as
     327             :   packages installed with different compilers.
     328             : - For security, it is good practice to make Spack [use the system's
     329             :   OpenSSL](https://spack.readthedocs.io/en/latest/getting_started.html#openssl)
     330             :   rather than allow it to install a new copy.
     331             : - To avoid reinstalling lots of system-provided packages with Spack, use the

     333             :   `spack install`). You can also install some of the dependencies with your
     334             :   system's package manager in advance, e.g., with `apt` or `brew`. If they are
     335             :   not picked up by `spack external find` automatically, register them with Spack
     336             :   manually. See the [Spack documentation on external

     338             :   for details.

     340             :   [LMod](https://github.com/TACC/Lmod). See the [Spack documentation on
     341             :   modules](https://spack.readthedocs.io/en/latest/module_file_support.html) for
     342             :   details.
     343             : - On macOS:
     344             :   - Brigand has an issue with AppleClang 13 when compiling tests (see
     345             :     https://github.com/edouarda/brigand/issues/274). Since it is header-only,
     346             :     you can simply clone the [Brigand repository](https://github.com/edouarda/brigand)
     347             :     instead of installing it with Spack.

     349             : ## Building Charm++ {#building-charm}
     350             : 
     351             : If you are not using a container, haven't installed Charm++ with Spack, or want
     352             : to install Charm++ manually for other reasons, follow the installation
     353             : instructions in the [Charm++ repository](https://github.com/UIUC-PPL/charm)
     354             : and in their [documentation](https://charm.readthedocs.io/en/latest/quickstart.html#installing-charm).
     355             : Here are a few notes:
     356             : 
     357             : - Once you cloned the [Charm++ repository](https://github.com/UIUC-PPL/charm),
     358             :   run `git checkout v7.0.0` to switch to a supported, stable release of
     359             :   Charm++.
     360             : - Apply the appropriate patch (if there is one) for the version from
     361             :   `${SPECTRE_ROOT}/support/Charm`. For example, if you have Charm++ v7.0.0
     362             :   then the patch will be `v7.0.0.patch`.
     363             : - Choose the `LIBS` target to compile. This is needed so that we can support the
     364             :   more sophisticated load balancers in SpECTRE executables.
     365             : - On a personal machine the correct target architecture is likely
     366             :   `multicore-linux-x86_64`, or `multicore-darwin-x86_64` on macOS. On an HPC
     367             :   system the correct Charm++ target architecture depends on the machine's
     368             :   inter-node communication architecture. It might take some experimenting to
     369             :   figure out which Charm++ configuration provides the best performance.
     370             : - Compile Charm++ with support for shared libraries by appending the option
     371             :   `--build-shared` to the `./build` command or pass `BUILD_SHARED=ON` to the
     372             :   CMake configuration (see the [Charm++ installation
     373             :   instructions](https://github.com/UIUC-PPL/charm#building-dynamic-libraries)).
     374             : - When compiling Charm++ you can specify the compiler using, for example,
     375             :   ```
     376             :   ./build LIBS ARCH clang
     377             :   ```

     379             : ## Building SpECTRE {#building-spectre}
     380             : 
     381             : Once you have set up your development environment you can compile SpECTRE.
     382             : Follow these steps:
     383             : 
     384             : 1. Create a build directory where you would like to compile SpECTRE. In the
     385             :    Docker container you could create, e.g., `/work/spectre-build`. It can be
     386             :    useful to add a descriptive label to the name of the build directory since
     387             :    you may create more later, e.g., `build-clang-Debug`. Then, `cd` into the
     388             :    build directory.
     389             : 2. Determine the location of your Charm++ installation. In the Docker container
     390             :    it is `/work/charm_7_0_0/multicore-linux-x86_64-gcc` for GCC builds and
     391             :    `/work/charm_7_0_0/mpi-linux-x86_64-smp-clang` for clang builds. For Spack
     392             :    installations you can determine it with
     393             :    `spack location --install-dir charmpp`. We refer to the install directory as
     394             :    `CHARM_ROOT` below.
     395             : 3. In your new SpECTRE build directory, configure the build with CMake:
     396             :    ```
     397             :    cmake -D CHARM_ROOT=$CHARM_ROOT SPECTRE_ROOT
     398             :    ```
     399             :    Add options to the `cmake` command to to configure the build, select
     400             :    compilers, etc. For instance, to build with clang you may run:
