       5             : # A Hitchhiker's Guide to Running SpECTRE {#beginners_guide}
       6             : 
       7             : \tableofcontents
       8             : 
       9             : SpECTRE can be a bit complicated to get started with, especially if you aren't
      10             : familiar with our core concepts of task-based parallelism and Template
      11             : Meta-Programming (TMP). However, <a
      12             : href="https://en.wikipedia.org/wiki/Phrases_from_The_Hitchhiker%27s_Guide_to_the_Galaxy#Don't_Panic">
      13             : Don't Panic</a>. This guide aims to get you introduced to running,
      14             : visualizing, editing, and then rebuilding SpECTRE to give you a feel for what
      15             : SpECTRE is all about, all on your own laptop! Hopefully by the end of this guide
      16             : you'll feel comfortable enough to look at other executables and maybe even
      17             : venture into the code itself!
      18             : 
      19             : ## Prerequisites
      20             : 
      21             : To start off, you'll need to obtain an environment to build and run SpECTRE in.
      22             : You could try and install all the dependencies yourself, but that is very
      23             : tedious and very error prone. Instead, we provide a
      24             : [Docker](https://docs.docker.com/get-docker/) container with all the
      25             : dependencies pre-installed for you to use. The container also has the SpECTRE
      26             : repository cloned in it already so you don't have to worry about getting it
      27             : yourself. To obtain the docker image, run
      28             : 
      29             : ```
      30             : docker pull sxscollaboration/spectre:demo
      31             : ```
      32             : 
      33             : Another program you will need for this tutorial is
      34             : [Paraview](https://www.paraview.org/download/) for visualizing the output. You
      35             : specifically will need version 5.10.1 for this tutorial.
      36             : 
      37             : If you'd like to use VSCode, the tutorial also has instructions for how to start
      38             : in VSCode as well.
      39             :

      45             : docker create --rm --name spectre_demo -p 11111:11111 \

     135             : ## Visualizing our BBH Coordinates
     136             : 
     137             : Now it's time to use Paraview to visualize the coordinates we use for our BBH
     138             : evolutions! SpECTRE will actually export the physical frame coordinates for
     139             : every executable we have because they are a really useful diagnostic to have. We
     140             : are just using the ExportCoordinates executable here so that you don't have to
     141             : run a BBH evolution on your laptop which probably wouldn't work because of
     142             : memory requirements.
     143             : 
     144             : Before we get to Paraview, we have to tell paraview how to actually use the
     145             : coordinates in the `Volume` `H5` file. To do this we have a tool called
     146             : `generate-xdmf` in our Python command-line interface. Inside the `runs`
     147             : directory where you have the `H5` files, run
     148             : 
     149             : ```
     150             : spectre generate-xdmf \
     151             :   --subfile-name element_data --output BBH_Coords \
     152             :   ExportTimeDependentCoordinates3DVolume*h5
     153             : ```
     154             : 
     155             : We output volume data per node so we append the
     156             : node number to each volume file we have. Since you're most likely running on a
     157             : laptop, you'll only be running on one node so you should only get one output
     158             : file for the volume. The `--subfile-name` argument is the group name inside the
     159             : `H5` file where the data is stored (groups can be checked by
     160             : `h5ls -r FILE_NAME`). `generate-xdmf` will generate a file called
     161             : `BBH_Coords.xmf`. Make sure to keep this `.xmf` file next to the volume file it
     162             : was generated from. It uses relative paths to find the volume file which means
     163             : if you move it, you won't be able to visualize anything.

     244             : ## Evolution of BBH Coordinates
     245             : 
     246             : Now that you are able to export and visualize our BBH domain coordinates at a
     247             : single time, let's make a small movie of the coordinates as they evolve! To do
     248             : this, we'll need to edit the input file `InputTimeDependent3D.yaml`. If you
     249             : aren't familiar with YAML, it's a file type that uses key-value pairs to create
     250             : actual objects in our C++ code. Feel free to experiment with keys and values in
     251             : our input files. If you're unsure about what a key or value should be, we offer
     252             : an easy way to check the options in the input file without running a whole
     253             : simulation. In your `/work/runs` directory, if you run
     254             : 
     255             : ```
     256             : ExportTimeDependentCoordinates3D \
     257             :   --input-file InputTimeDependent3D.yaml --check-options
     258             : ```
     259             : 
     260             : the `--check-options` will tell the executable to parse the input file and exit
     261             : immediately after. If you made a typo, or added an incorrect key/value, a list
     262             : of the available keys and their associated values will be printed.
     263             : 
     264             : To change the number of times we output the coordinates, we'll need to go to the
     265             : `%EventsAndTriggers:` block of the input file. This block is mainly where we
     266             : specify which quantities we want to observe in a simulation or where we
     267             : "Trigger" a specific "Event" to happen. (For more info on `%EventsAndTriggers`,
     268             : see the \ref tutorial_events_and_triggers tutorial.) Currently in this input
     269             : file we only have one Trigger/Event pair. The %Trigger is `TimeCompares:` and
     270             : the %Event is `Completion`. To have the simulation run longer, change the
     271             : `Value:` under `TimeCompares:` to something larger. If you look at the
     272             : `Evolution:` block above the `%EventsAndTriggers:` block, you'll see that the
     273             : initial time step is `0.5`. The way this executable is set up, the coordinates
     274             : will be exported every time step. So set the final time `Value:` under
     275             : `TimeCompares:` to some larger multiple of `0.5` so that you'll have the
     276             : coordinates at a bunch of different times (a final time of `20` is reasonable.
     277             : Depending on how many cores you run on this should take a couple minutes).
     278             : 
     279             : Then, run the executable just like you did above (remember to move or delete the
     280             : existing `H5` files), run `generate-xdmf`, and open it in Paraview and apply
     281             : some filters of your choice. We recommend using a `Slice` filter with the normal
     282             : pointing in the `-z` direction. This is because our BBH domain rotates about the
     283             : `z` axis. Now, in the top bar of Paraview, you should see a "Play" button that
     284             : looks like a sideways triangle (see the second image in the \ref open_xmf
     285             : section). If you click this, Paraview will step through all the timesteps in the
     286             : output files and you'll be able to see the domain rotate a bit!
     287             : 
     288             : Next, we encourage you to play with the other inputs that control how the domain
     289             : evolves over time. These options are housed in the
     290             : 
     291             : ```yaml
     292             : DomainCreator:
     293             :   BinaryCompactObject:
     294             :     ...
     295             :     TimeDependentMaps:
     296             :       ExpansionMap:
     297             :         ...
     298             :       RotationMap:
     299             :         ...
     300             :       SizeMapA:
     301             :         ...
     302             :       SizeMapB:
     303             :         ...
     304             : ```
     305             : 
     306             : block of the input file. Since this tutorial is more about running the code, we
     307             : won't go into too much detail about each option. However, in general:
     308             : 
     309             : 1. `ExpansionMap` is a global map (all parts of the domain) that controls the
     310             :    separation between the excision surfaces
     311             : 2. `RotationMap` is a global map that controls how the excision spheres rotate
     312             :    about each other
     313             : 3. `SizeMap` is a local map only around the excision spheres (not in the wave
     314             :    zone) that control the compression of grid points.
     315             : 
     316             : Play around with these values! You may get an error if you put something that's
     317             : too unphysical, but this is a fairly consequence-free playground for you to
     318             : explore so just try a different value.

     322             : ## Exploring DG+FD
     323             : 
     324             : Now that you are able to run, and visualize SpECTRE, let's explore a feature
     325             : that is fairly unique to SpECTRE and is really powerful for handling
     326             : discontinuities and shocks in our simulations. We call this feature `DG+FD`
     327             : (it's also sometimes referred to as just `subcell`).
     328             : 
     329             : ### Description of DG+FD
     330             : 
     331             : `FD` is the usual finite difference you are used to. All of the BSSN codes use
     332             : finite difference for solving Einstein's equations. FD is very good at capturing
     333             : shocks and discontinuities and is a very robust method, making it well suited
     334             : for hydro problems and other systems that have shocks and discontinuities.
     335             : 
     336             : `DG` stands for Discontinuous Galerkin. DG is a spectral method for representing
     337             : a solution on a grid, meaning that instead of taking the difference between the
     338             : function value at two points to get the derivative, it uses known basis
     339             : functions to represent the solution. Then the derivative can be known
     340             : analytically and you only need to supply the coefficients for the basis. DG
     341             : works best for representing smooth solutions; ones with very few shocks and
     342             : discontinuities (like GR in vacuum). This makes DG much faster than FD for
     343             : smooth solutions.
     344             : 
     345             : In SpECTRE, we combine these two different methods into one system to take
     346             : advantage of the strengths of each. When we have a solution that is smooth in
     347             : some parts of the domain, but has shocks in other parts, using only one of these
     348             : methods has disadvantages. If we only used DG, we wouldn't be able to resolve
     349             : the shocks very well driving the errors up a lot. If we only used FD, we'd be
     350             : able to represent the solution well, but it would be computationally
     351             : inefficient. So we combine DG+FD so that we only do DG in the parts of the
     352             : domain where the solution is smooth, and switch to FD in parts where there may
     353             : be a shock or discontinuity. The algorithm for switching between DG and FD is
     354             : explained in this image.
     355             : 
     356             : \image html dg_fd_schematic.png "Scheme for switching between DG and FD (credit: Nils Deppe)"
     357             : 
     358             : If you'd like to learn more about how SpECTRE implements its DG+FD scheme, you
     359             : can read [the paper](https://arxiv.org/abs/2109.11645) on the ArXiv.
     360             : 
     361             : ### Running the Kuzmin Problem
     362             : 
     363             : To demonstrate DG+FD, we will be evolving the \link
     364             : ScalarAdvection::Solutions::Kuzmin Kuzmin \endlink problem using the
     365             : `EvolveScalarAdvection2D` executable. This is a simple test problem that
     366             : rotates a set of geometric shapes with uniform angular velocity, which can be
     367             : used to evaluate how well a numerical code can handle discontinuities stably
     368             : over time. Inside the container make a new directory `/work/runs2` where you
     369             : will run it. Also copy the default input file in
     370             : `/work/spectre/tests/InputFiles/ScalarAdvection/Kuzmin2D.yaml` to this new
     371             : `/work/runs2` directory.
     372             : 
     373             : ### Changing the Default Input File
     374             : 
     375             : The default input file has very low resolution so we'll need to crank that up a
     376             : bit. The way to do this is to change the initial refinement levels and initial
     377             : number of grid points which are located in
     378             : 
     379             : ```yaml
     380             : DomainCreator:
     381             :   Rectangle:
     382             :     ...
     383             :     InitialRefinement: [x, y]
     384             :     InitialGridPoints: [x, y]
     385             : ```
     386             : 
     387             : `InitialRefinement:` represents how many times we split a `Block` in half in
     388             : order to create `Element`s, which are the fundamental units of our domain. So an
     389             : initial refinement of `[1, 1]` means we split a single Block into 4 elements
     390             : (split in half once in each direction). For an initial refinement of `[2, 2]` we
     391             : first do 1 refinement like before, and then split each of the resulting 4
     392             : elements in half again in each direction, resulting in 16 total Elements. To
     393             : determine the total number of Elements for a given refinement (same in all
     394             : directions), just do $2^{\mathrm{Dim * Refinement}}$. If you're confused by
     395             : the terminology we use to describe the domain, we have a \ref domain_concepts
     396             : guide that explains all terms related to our domain.
     397             : 
     398             : `InitialGridPoints` represents the number of grid points per dimension in each
     399             : Element after the final refinement has been applied. So if we had an initial
     400             : refinement of `[2, 2]` like above and then initial grid points `[3, 3]` in each
     401             : Element, we'd have a total of 9x16=144 grid points.
     402             : 
     403             : As for actual numbers to use, you can experiment to see what gives good,
     404             : well-resolved results. You'll definitely need more refinement than the default
     405             : input file, but since refinement scales exponentially, this can become very
     406             : expensive very quickly. On a laptop, you probably shouldn't go higher than
     407             : refinement `[6, 6]`. As for grid points, this will depend on how much refinement
     408             : you have. If you have a ton of small elements, you won't need too many grid
     409             : points to resolve the solution; something like `[4, 4]` would work. If you don't
     410             : have a lot of refinement, you may want more grid points if you still want to
     411             : resolve your solution. For a DG scheme, increasing the number of grid points (p
     412             : refinement) reduces the numerical error exponentially where the solution is
     413             : smooth, so computational resources are used more effectively. However, to
     414             : resolve shocks and discontinuities we have to refine the domain into more and
     415             : smaller elements instead (h refinement). Striking the most effective balance
     416             : between h and p refinement in different parts of the domain is the job of an
     417             : adaptive mesh refinement (AMR) algorithm.
     418             : 
     419             : The default input file only runs for a few time steps so we'll want to make this
     420             : run longer so we can actually see some evolution. From the documentation of the
     421             : \link ScalarAdvection::Solutions::Kuzmin Kuzmin \endlink system, the solution
     422             : will rotate with an angular velocity of `1.0` (in code units). Thus, to do a
     423             : full orbit, it will take `6.28` code units of time. In the `%EventsAndTriggers:`
     424             : block of the input file, we see that the `Completion` event is triggered by the
     425             : `Slabs` trigger. We could, in theory, calculate out how many slabs `6.28` code
     426             : units is using the time step, but that's super tedious. Instead let's trigger
     427             : completion using the `TimeCompares` trigger instead. We used this before when
     428             : exporting the BBH coordinates, so just copy over the yaml block and change the
     429             : `Value:`.
     430             : 
     431             : Your final `%EventsAndTriggers:` block should look something like this:
     432             : 
     433             : ```
     434             : EventsAndTriggers:
     435             :   ? TimeCompares:
     436             :       Comparison: GreaterThanOrEqualTo
     437             :       Value: 6.28
     438             :   : - Completion
     439             :   ...
     440             : ```
     441             : 
     442             : Now you should be ready to run the executable and get some output. Here, you
     443             : will almost definitely benefit by running this on many cores by adding the
     444             : `++ppn N` flag to the command you use to run the executable. Since we use lots
     445             : of smaller elements, we distribute these over the available resources via a
     446             : \link domain::BlockZCurveProcDistribution space filling curve \endlink to speed
     447             : things up.
     448             : 
     449             : ```
     450             : EvolveScalarAdvection2D --input-file Kuzmin2D.yaml ++ppn 4
     451             : ```
     452             : 
     453             : ### Visualizing the Kuzmin Problem
     454             : 
     455             : Once your run finishes, extract the volume data with `generate-xdmf` using
     456             : 
     457             : ```
     458             : spectre generate-xdmf \
     459             :   --file-prefix ScalarAdvectionKuzmin2DVolume \
     460             :   --subfile-name VolumeData --output kuzmin_problem
     461             : ```
     462             : 
     463             : (Note that the `subfile-name` is different than before because it was different
     464             : in the input file) and load it into Paraview once again. We are only interested
     465             : in the quantity `U` which is the scalar field we were evolving. You can uncheck
     466             : any other boxes. So now, instead of coordinates on your screen, you should see a
     467             : large square colored by the solution profile described in the \link
     468             : ScalarAdvection::Solutions::Kuzmin Kuzmin \endlink system. You should also
     469             : notice that there are smaller squares that don't touch each other in the middle
     470             : of the domain and on the edges there are large sections that are continuous.
     471             : These are the FD and DG grids, respectively. If you go to the top bar in
     472             : Paraview and change how you view the grid to `Surface With Edges`, this will
     473             : become even more apparent.
     474             : 
     475             : You will notice that the FD grid is mostly around where the interesting features
     476             : are in the solution profile; the cylinder with a wedge cut out, the cone, and
     477             : the hump. And then the DG grid is mostly where the solution should be zero
     478             : towards the boundary of the domain (i.e. the very smooth part). So right from
     479             : the start, you can see that we are saving computational effort by only doing the
     480             : expensive, yet robust, method (FD) where it is necessary and the efficient
     481             : method (DG) everywhere else where the solution is smooth.
     482             : 
     483             : Now hit the "Play" button in the top bar of Paraview and watch the solution
     484             : evolve. You'll notice that the elements in the domain switch back and forth
     485             : between FD and DG. They do so in such a way that the elements will switch to FD
     486             : when an interesting feature enters the element and then switch back to DG once
     487             : the feature leaves. In this way, we are able to actually track shocks and
     488             : discontinuities in real time in our solution by where the code switches to using
     489             : FD instead of DG. This is extremely useful for expensive GRMHD simulations where
     490             : we only want to do FD at a shock boundary, yet that shock boundary is moving
     491             : through the domain. We are able to dynamically track this shock and resolve it
     492             : well with FD, then switch back to DG after the shock passes through and the
     493             : solution has settled down again.
     494             : 
     495             : A pretty cool filter you can add is `Warp By Scalar`. In the left panel, choose
     496             : the solution variable `U` as the scalar to use and hit `Apply`. In the viewing
     497             : panel there should be a `2D` or `3D` button that you can toggle to make the view
     498             : 3D. Once you do that you should be able to see that the height of the feature is
     499             : your solution `U`. If you change the grid to `Surface With Edges` you can see
     500             : the FD or DG grids warp with the solution. And if you hit "Play" it'll rotate
     501             : around and you'll see the features moving in 3D! (Don't worry if you can't find
     502             : this filter. Not all versions of Paraview may have it.)
     503             : 
     504             : We encourage you to play around with the refinement and grid points before the
     505             : next section to get a feel for how each changes the runtime and accuracy of
     506             : solution.

     508             : ## Editing the Kuzmin System
     509             : 
     510             : Hopefully now you feel comfortable enough running SpECTRE that you
     511             : can get the default input file for the pre-built executables, edit it, and run
     512             : it. Now we are going to try our hand at actually editing some code in SpECTRE
     513             : and then building SpECTRE. We're going to stick with the \link
     514             : ScalarAdvection::Solutions::Kuzmin Kuzmin \endlink system and add a new feature
     515             : to the solution profile!
     516             : 
     517             : You can find the files for the Kuzmin system at
     518             : `/work/spectre/src/PointwiseFunctions/AnalyticSolutions/ScalarAdvection/
     519             : Kuzmin.?pp`.
     520             : In the `hpp` file, you'll see a lot of Doxygen documentation and then the actual
     521             : Kuzmin class. The only function that you will need to care about is
     522             : 
     523             : ```cpp
     524             : template <typename DataType>
     525             : tuples::TaggedTuple<ScalarAdvection::Tags::U> variables(
     526             :     const tnsr::I<DataType, 2>& x, double t,
     527             :     tmpl::list<ScalarAdvection::Tags::U> /*meta*/) const;
     528             : ```
     529             : 
     530             : All of our analytic solutions have a function similar to this that will set the
     531             : value corresponding to the tag in the return type. If you're unfamiliar with
     532             : tags in SpECTRE, you can look at these sections for an explanation, \ref
     533             : databox_a_taggedtuple_databox and \ref databox_a_proper_databox. However, it's
     534             : basically just a fancy way of doing a compile-time key/value pair. The tag is
     535             : the key, and the value is whatever you want it to be. In our case, the value is
     536             : a tensor, representing the solution.
     537             : 
     538             : The definition of this function in the `cpp` file is where you will be editing
     539             : the actual Kuzmin solution. Towards the bottom of this function, there is a
     540             : `for` loop that sets the solution at every grid point. This is where you will
     541             : add in a new feature to the solution.
     542             : 
     543             : You can pick any feature you want to add, so long as it's inside the domain
     544             : bounds of `[0,1]x[0,1]` and centered around `(0.75, 0.5)`. This is because of
     545             : how the kuzmin solution is set up with existing features at `(0.25, 0.5); (0.5,
     546             : 0.25); (0.5, 0.75)`. If you're having trouble thinking of a feature to add try
     547             : one of the following features:
     548             : 
     549             : - Square centered at `(0.75, 0.5)` with solution value `1.0`
     550             :   - Side length `0.1` (any larger and it might interfere with the other
     551             :     features)
     552             :   - Circle of radius `0.045` centered on the square with value `0.0`
     553             : - Triangle centered at `(0.75, 0.5)` with one corner facing in the `+x`
     554             :   direction with solution value `1.0`
     555             : - Square centered at `(0.75, 0.5)`
     556             :   - Side length `0.1` (any larger and it might interfere with the other
     557             :     features)
     558             :   - Left half of the square has value `1.0` and right half of the square has
     559             :     value `0.5`
     560             : 
     561             : \note The more detailed you make your feature, the more resolution you will need
     562             : to resolve it.
