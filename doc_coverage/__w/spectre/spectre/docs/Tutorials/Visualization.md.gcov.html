       9             : ### Building an Executable from an Existing Source File
      10             : 
      11             : SpECTRE source files for evolution executables are located in
      12             : `src/Evolution/Executables`. Executables can be compiled by running the command
      13             : `make EXECUTABLE_NAME` where `EXECUTABLE_NAME` is the name of the executable
      14             : as defined in the `CMakeLists.txt` file located in the same directory as the
      15             : source file. For example, to compile the executable that evolves a scalar wave
      16             : using a three-dimensional domain, navigate to `$SPECTRE_HOME/build/`, then
      17             : one runs the command: `make EvolveScalarWave3D`, which then results
      18             : in an executable of the same name in the `bin` directory of the user's
      19             : build directory.
      20             : 
      21             : ### Running an Evolution Executable
      22             : 
      23             : Each SpECTRE executable reads a user-provided YAML file that specifies the
      24             : runtime settings of the evolution. This is where options such as the Domain,
      25             : AnalyticSolution, TimeStepper, etc are supplied. Example input files are kept
      26             : in `tests/InputFiles`, and are written to provide a functional base input file
      27             : which the user can then modify as desired. Copy the executable and YAML file
      28             : to a directory of your choice. The YAML file is then passed as an argument to
      29             : the executable using the flag `--input-file`. For example, for a scalar wave
      30             : evolution, run the command:
      31             : `./EvolveScalarWave3D --input-file PlaneWave3D.yaml`.
      32             : By default, the example input files do not produce any output. This can be
      33             : changed by modifying the options passed to `EventsAndTriggers` or
      34             : `EventsAndDenseTriggers`:
      35             : 
      36             : \snippet PlaneWave1DObserveExample.yaml observe_event_trigger
      37             : 
      38             : This will observe the norms of the errors in the system at times
      39             : \f$0.0\f$ and \f$1.0\f$ and the field data at the start of every 50th
      40             : slab.  A successful observation will result in the creation of H5
      41             : files whose names can be specified in the YAML file under the options
      42             : `VolumeFileName` and `ReductionFileName`. One volume data file will be
      43             : produced from each Charm++ node that is used to run the
      44             : executable. Each volume data file will have its corresponding node
      45             : number appended to its file name.  Visualization of the volume data
      46             : will be described in the next section.
      47             : 
      48             : ### 3D Data Volume Data In ParaView
      49             : 
      50             : A SpECTRE executable with observers produces volume and/or reduced data h5
      51             : files. An XDMF file must be created from the volume data in order to do
      52             : visualization using ParaView. To this end we provide the tool
      53             : `generate-xdmf` in the Python command-line interface. Run it in your build
      54             : directory as `spectre generate-xdmf`. It
      55             : takes two required arguments which are passed to `--subfile-name`, and
      56             : `--output`. The
      57             : argument passed to `--subfile-name` is the name of the subfile inside the H5
      58             : volume data file where data is stored. In the above example, `--subfile-name`
      59             : would be `VolumePsiPiPhiEvery50Slabs`. The argument passed to `--output` is the
      60             : desired .xmf file name, also without filename extension. Use `--help` to see a
      61             : further description of possible arguments.
      62             : 
      63             : Open the .xmf file in ParaView and select the `Xdmf Reader`, *not* the version 3
      64             : readers. On the left hand side of the main ParaView window is a section named
      65             : `Properties`, here you must click the highlighted `Apply` button. ParaView will
      66             : now render your volume data. If you only wish to visualize a few datasets out of
      67             : a large set, we recommended unchecking the boxes for the datasets you wish to
      68             : ignore under `Point Arrays` before clicking `Apply`. Also, don't forget to check
      69             : which dataset is being used for the color under `Coloring`. The default won't
      70             : necessarily be the dataset you want.
      71             : 
      72             : \note ParaView cannot understand `.xmf` files when subfile names have colons in
      73             : them. Please avoid subfile names like `My::Subfile:Name`.
      74             : 
      75             : ### Helpful ParaView Filters
      76             : 
      77             : Here we describe the usage of filters we've found to better visualize our data.
      78             : Feel free to contribute to this section!
      79             : 
      80             : #### Removing Mesh Imprinting
      81             : You may notice what appears to be mesh imprinting on the data. The imprinting
      82             : effect can be removed by applying the `Tetrahedralize` filter. To apply the
      83             : filter select the `Filters` menu item, then `Alphabetical` and finally
      84             : `Tetrahedralize`.
      85             : 
      86             : #### Creating Derived Volume Data
      87             : New volume data can be created from the existing volume data using the
      88             : `Calculator` filter. In the `Calculator`'s text box, input a numerical
      89             : expression in terms of existing datasets evaluating to the desired
      90             : quantity. For example, a vector-valued velocity dataset can be created
      91             : from three scalar velocity component datasets using the expression
      92             : `velocity_x * iHat + velocity_y * jHat + velocity_z * kHat` and hitting
      93             : the `Apply` button. By default, this will create a new dataset `Result`.
      94             : The name of the new dataset can be changed by changing the name provided
      95             : to `Result Array Name` above the `Calculator` text box.
      96             : 
      97             : #### Visualizing Vector Fields Derived From Scalar Fields
      98             : Use the `Calculator` filter described above in "Creating Derived Volume Data"
      99             : to create a new vector-valued dataset. Once this is created, use the `Glyph`
     100             : filter and set the `Active Attributes` to the vector you wish to visualize.
     101             : Make sure that `Scale Mode` is set to `vector`.
     102             : 
     103             : ### Visualizing and Extracting Dat Files
     104             : 
     105             : Global quantities such as error norms are stored in `h5::Dat` subfiles in the
     106             : reduction/global HDF5 file. The command-line endpoint `spectre plot-dat`
     107             : can be used to plot quantities, and `spectre extract-dat` can be used
     108             : to extract them into space-separated text files. These text
     109             : files can then be read with other plotting programs or viewed in an editor.
     110             : 
     111             : ### Reproducibility of Results
     112             : 
     113             : Being able to reproduce the results of simulations is important for scientific
     114             : integrity and accuracy. To this end, SpECTRE encodes the source tree,
     115             : environment variables, and a record of build information (e.g., the third-party
     116             : library versions) into the executables. From most executables these can be
     117             : retrieved using the flags `--dump-source-as filename_without_extension`,
     118             : `--dump-paths`, `--dump-build-info`, and `--dump-environment`. The flag
     119             : `--dump-only` will cause the program to exit after the information has been
     120             : dumped. You can use the `--help` flag to get more details on what flags are
     121             : available. The flag `--dump-paths` will print to screen the environment
     122             : variables `PATH`, `CPATH`, `LD_LIBRARY_PATH`, `LIBRARY_PATH`, and
     123             : `CMAKE_PREFIX_PATH` at link time. The flag `--dump-build-info` will print to
     124             : screen the contents of the `BuildInfo.txt` file generated by CMake. The flag
     125             : `--dump-environment` will print to screen the output of `printenv` at link
     126             : time. Finally, the flag `--dump-source-as filename_without_extension` will dump
     127             : the archive `filename_without_extension.tar.gz`, which contains the entire
     128             : source tree at link time. The combined above information should be enough to
     129             : reproduce simulations results to roundoff error as long as the third party
     130             : libraries used to build the executable have not changed on the system the
     131             : simulation was run on.
     132             : 
     133             : Often only data from the simulation is retained, not the entire executable. In
     134             : order to make simulations reproducible from only the HDF5 output, any HDF5 files
     135             : written with SpECTRE's HDF5 wrappers contain the info about the run, output of
     136             : `printenv` at link time, the contents of `BuildInfo.txt`, and an archive
     137             : of the source tree. These are inside the `header.hdr` group with names
     138             : `SimulationInfo`, `EnvironmentVariablesAtCompilation`, `BuildInfo`, and
     139             : `src.tar.gz`. The source archive can be extracted by running the command
     140             : ```
     141             : h5dump -d /src.tar.gz -b LE -o src.tar.gz /path/to/hdf5/file.h5
     142             : ```
     143             : You will need to make sure the argument to `-b` is the same endianness as your
     144             : computer. In the above example little endian is used.
